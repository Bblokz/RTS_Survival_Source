Runtime tow relationship state
In addition to their edit-default settings, both tow components must store runtime state describing the active tow relationship. This runtime state is the source of truth for whether an actor is currently towing another actor or is currently being towed. This state must not be inferred from the currently active ability, the current command in the queue, or the current command card layout.

The following runtime state must be added:

For UVehicleTowComponent:

TWeakObjectPtr<AActor> M_TowedActor;

TWeakObjectPtr<UTowedActorComponent> M_TowedActorComp;

ETowActorAbilitySubtypes M_CurrentTowSubtype = ETowActorAbilitySubtypes::None;

For UTowedActorComponent:

TWeakObjectPtr<AActor> M_TowingActor;

TWeakObjectPtr<UVehicleTowComponent> M_TowingVehicleTowComp;

This stored runtime state must be updated immediately when towing starts and cleared immediately when towing ends.
IsTowFree() on both components must use this runtime relationship state instead of relying on command or ability state.

This means:

a towing vehicle is considered not tow free when its UVehicleTowComponent has a valid current towed actor/component and a non-None current tow subtype;

a towed actor is considered not tow free when its UTowedActorComponent has a valid towing actor/component.

This runtime subtype is also what must be used later to determine whether the currently attached actor is a vehicle or a team weapon during detach handling.

When towing or detaching requires logic on the physical team weapon actor, it is the responsibility of the ATeamWeaponController to forward that logic to its controlled ATeamWeapon. This includes, but is not limited to, attaching and detaching the team weapon actor, setting its relative transform at the tow socket, resetting turret yaw and pitch, enabling or disabling turret behaviour, playing wheel animations, and any other weapon-specific visual or movement updates needed while towing.

Immediate internal cargo transfer for team-weapon towing
When a team weapon begins being towed, transferring its alive squad units into the towing tank’s cargo must use a dedicated immediate/internal cargo transfer path. This must not use the regular queued EnterCargo command flow.

The reason for this is that the tow attach sequence must complete deterministically as one internal towing operation. Regular queued boarding would introduce command queue progression, asynchronous unit-level command completion, and timing dependencies that are not desirable for this towing flow.

Therefore, if existing cargo functionality is not already suitable for immediate internal transfer, a dedicated internal API must be added for this case. This internal cargo transfer path must:

validate cargo capacity for the number of currently alive squad units before transfer begins;

immediately move valid squad units into the towing tank’s cargo without requiring per-unit queued EnterCargo commands;

avoid relying on normal unit command completion callbacks to finish the tow sequence;

be safe to use after the team weapon controller has already been forced to idle;

allow the team weapon towing sequence to continue immediately once the squad has been placed into cargo.

This internal path is only for the towing flow of team weapons and is separate from normal player-issued cargo boarding behaviour.

Explicit team-weapon towed state
The ATeamWeaponController must have an explicit runtime state indicating that the team weapon is currently being towed. This must be represented either by adding a dedicated ETeamWeaponState::Towed state or by adding a dedicated boolean such as bM_IsBeingTowed. A dedicated state enum entry is preferred because towing is a meaningful top-level team-weapon state and should be treated the same way as packing, deploying, moving, and abandoned.

This state is required because the existing team-weapon controller contains logic for packing, deploying, moving, rotation requests, dig-in handling, auto-deploy on idle, guard repositioning, crew snapping, and out-of-range repositioning. While the team weapon is being towed, these flows must not continue running as if the team weapon were still operating independently.

While the team weapon is in the towed state, the controller must gate or early-return from all logic that would otherwise try to operate the weapon normally, including at least:

auto-deploy-on-idle logic;

deploy or pack transitions;

rotation request ticking and completion handling where not explicitly intended;

turret out-of-range reposition flow;

issuing crew move orders to weapon crew positions;

moving guards to random guard positions;

any other logic that assumes the team weapon and its crew are present in the world as a normal deployed or packed team weapon.

Entering the towed state must happen immediately when the tow attach is completed.
Leaving the towed state must happen immediately when the detach is completed.
