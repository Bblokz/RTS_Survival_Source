# World Campaign Fog of War Design

## Goal

The world campaign Fog of War system should create a Civilization-style cloud layer over the campaign map.
This system is for the world campaign map, not for regular RTS gameplay. It should be driven by the campaign graph and updated only at predictable moments, such as after the world is generated, after the player progresses, or after a turn/world-state change.
The visual output is a generated mask texture used by the large cloud cube material above the map.
The gameplay source of truth is not the texture. The gameplay source of truth is the world campaign state, the anchor graph, and the FOW state stored on participating actors.
Core rule:
**`AAnchorPoint` and `AConnection` define what can be discovered. `AWorldFowManager` calculates the current FOW state. `UWorldMapFowComponent` stores the local actor state. The generated mask texture only visualizes that state.**
---

# Existing campaign-map concepts used by this system

## `AWorldPlayerController`

`AWorldPlayerController` remains the player-facing owner of world map input.

It already handles:

* world generator setup;
* world menu setup;
* world campaign generation/load flow;
* cursor hit detection;
* player outliner ticking;
* primary click routing;
* camera movement.

The FOW system should integrate into this flow without replacing the existing click handlers.

`AWorldPlayerController` should spawn the new `AWorldFowManager` during `BeginPlay`, after the campaign world has been generated or restored into the level.

## `AGeneratorWorldCampaign`

`AGeneratorWorldCampaign` remains responsible for generating the world campaign layout.

It owns or exposes the generated campaign state through:

* cached anchors;
* cached connections;
* player HQ anchor;
* enemy HQ anchor;
* enemy item placements by anchor key;
* neutral item placements by anchor key;
* mission placements by anchor key;
* generated connection topology;
* anchor hop data and derived placement data.

`AWorldFowManager` should read this data after generation is complete.

`AWorldFowManager` should not duplicate campaign generation rules, placement rules, mission placement logic, or enemy placement logic.

## `AAnchorPoint`

`AAnchorPoint` is the main campaign-map location unit.

In this design, every explorable or visible campaign location is anchor-driven.

`AWorldFowManager` should use `AAnchorPoint` as the primary graph element for:

* visible locations;
* explorable frontier locations;
* player HQ start reveal;
* connected campaign routes;
* promoted world map objects;
* reveal mask stamping.

## `AConnection`

`AConnection` is the graph route between anchors.

The FOW system should use connections to reveal or hint paths between anchors.

This prevents the cloud reveal from looking like disconnected circular holes. If one anchor is visible and another is explorable, the route between them should usually receive an explorable/path hint.

## `AWorldMapObject`

`AWorldMapObject` is the base actor for promoted world campaign objects.

Relevant subclasses include:

* `AWorldEnemyObject`;
* `AWorldMissionObject`;
* `AWorldNeutralObject`;
* `AWorldPlayerObject`.

`AWorldMapObject` already has the concept of an owning anchor and an anchor key. The FOW system should use that relationship as the main link between a promoted world object and its campaign location.

Default behavior:

* the promoted object follows the FOW state of its owning `AAnchorPoint`;
* special POI objects can override this by becoming `POIVisible`.

---

# New enum: `EWorldMapFowState`

Create a world campaign FOW enum named:

`EWorldMapFowState`

Required values:

* `NotVisible`;
* `Explorable`;
* `Visible`;
* `POIVisible`.

This enum should be owned by the world campaign FOW system and used by `UWorldMapFowComponent`.

---

# FOW state behavior

## `NotVisible`

Meaning:

The actor or location is hidden under the world campaign cloud layer.

Gameplay behavior:

* primary-click interaction is not allowed;
* object cannot be outlined.

Visual behavior:

* the cloud remains covered in this area;
* the actor may be hidden, faded, or visually blocked by the cloud;
* no world-map hover feedback should be shown.

---

## `Explorable`

Meaning:

The actor or location is not fully revealed, but it is available as part of the current exploration frontier.

An explorable location is usually adjacent to a visible anchor.

Gameplay behavior options:

* primary-click interaction is allowed;
* object can be outlined.

Visual behavior:

* cloud can be thinned around the actor or its owning anchor;
* connected route can be hinted;
* actor can appear as a frontier option.

## `Visible`

Meaning:

The actor or location is fully revealed.

Gameplay behavior options:

* primary-click interaction is allowed;
* object can be outlined.

Visual behavior:

* cloud is cleared or strongly reduced around the actor or owning anchor;
* actor is presented as fully available on the world map.

Important rule:

`Visible` should not imply extra interaction types by itself. It only guarantees primary-click interaction and outline eligibility.

Any deeper gameplay behavior should remain owned by the existing world map systems and click handlers.

---

## `POIVisible`

Meaning:

The actor is visible as a landmark above or through the cloud, but the owning anchor is not necessarily revealed.

Gameplay behavior:

* object can be outlined.

Visual behavior:

* actor can appear as a distant landmark;
* tall building top, tower, antenna, beacon, or silhouette can be visible;
* cloud may show a small POI break, glow, or silhouette around it.

Important rule:

`POIVisible` does not allow primary-click interaction by default.

It only allows outline behavior.

`POIVisible` should not automatically reveal the owning `AAnchorPoint`.

---

# FOW behavior table

| FOW State    | Primary-click interaction | Outline |
| ------------ | ------------------------: | ------: |
| `NotVisible` |                        No |      No |
| `Explorable` |                       Yes |     Yes |
| `Visible`    |                       Yes |     Yes |
| `POIVisible` |                        No |     Yes |

This table should be implemented once in the FOW component/helper layer.

Other systems should ask the FOW component or interface helpers instead of duplicating this logic.

---

# New component: `UWorldMapFowComponent`

Create a new component named:

`UWorldMapFowComponent`

Designer-facing component name:

`WorldMapFow`

Designers are expected to add this component in derived Blueprints.

The C++ base classes should not create the component automatically.

## Component responsibility

`UWorldMapFowComponent` stores the local FOW state for its owning actor.

It should expose helper functions for:

* current `EWorldMapFowState`;
* whether the current state allows primary-click interaction;
* whether the current state allows outlining;
* whether the current state contributes to the visible mask channel;
* whether the current state contributes to the explorable mask channel;
* whether the current state contributes to the POI mask channel;
* reveal radius for the current state;
* reveal falloff for the current state;
* connection corridor width for the current state;
* POI reveal settings.

The component should be the single place where enum-to-behavior mapping lives.

## Per-state settings

The component should allow designer tuning per FOW state.

Suggested settings:

| Setting                   | Purpose                                                  |
| ------------------------- | -------------------------------------------------------- |
| Reveal Radius             | Radius used when this actor contributes to the mask.     |
| Reveal Falloff            | Soft edge around the reveal stamp.                       |
| Connection Corridor Width | Width used by connection/path stamps.                    |
| Writes Visible Mask       | Whether the state contributes to the visible channel.    |
| Writes Explorable Mask    | Whether the state contributes to the explorable channel. |
| Writes POI Mask           | Whether the state contributes to the POI channel.        |
| Allows Primary Click      | Derived from enum behavior.                              |
| Allows Outline            | Derived from enum behavior.                              |
| Can Become POI Visible    | Enables special POI behavior.                            |
| POI Reveal Radius         | Smaller reveal radius for POI landmarks.                 |
| POI Reveal Falloff        | Softness of POI reveal area.                             |

Primary-click and outline behavior should normally come from the enum mapping, not from unrelated per-actor ad hoc logic.

---

# New interface: `IWorldFowParticipant`

Create a new interface named:

`IWorldFowParticipant`

Any actor that participates in world campaign FOW should implement this interface.

Expected implementers:

* `AWorldMapObject`;
* `AWorldEnemyObject`;
* `AWorldMissionObject`;
* `AWorldNeutralObject`;
* `AWorldPlayerObject`;
* `AAnchorPoint`;
* `AConnection`;

## Interface purpose

The interface allows systems such as `AWorldPlayerController`, `UPlayerWorldOutliner`, and `AWorldFowManager` to query FOW state without knowing the exact actor subclass.

The interface should provide helpers for:

* getting the cached `UWorldMapFowComponent`;
* checking whether the actor can be outlined;
* checking whether the actor can receive primary-click interaction.

## `GetFowComponent`

The interface should expose a helper that returns the cached FOW component.

This should not repeatedly call `FindComponentByClass`.

Each actor should cache the component during `PostInitializeComponents`.

## `GetCanBeOutlined`

The interface should expose a helper named:

`GetCanBeOutlined`

This helper should ask the cached `UWorldMapFowComponent` whether the current enum state allows outlining.

The interface should not duplicate the enum logic.

Conceptually:

`IWorldFowParticipant::GetCanBeOutlined`

delegates to:

`UWorldMapFowComponent::GetCanBeOutlinedForCurrentState`

## `GetCanPrimaryClickInteract`

The interface should also expose a helper for primary-click gating.

This helper should ask the cached `UWorldMapFowComponent` whether the current enum state allows primary-click interaction.

Conceptually:

`IWorldFowParticipant::GetCanPrimaryClickInteract`

delegates to:

`UWorldMapFowComponent::GetCanPrimaryClickInteractForCurrentState`

---

# FOW component caching on actors

Participating actors should search for their `UWorldMapFowComponent` during `PostInitializeComponents`.

The reference should then be cached on the actor as a weak pointer.

Recommended pattern:

1. Designer adds `WorldMapFow` component in the derived Blueprint.
2. C++ actor runs `PostInitializeComponents`.
3. Actor searches for `UWorldMapFowComponent`.
4. Actor caches the reference.
5. `IWorldFowParticipant` helpers use the cached reference.
6. Repeated click/outline checks do not search components again.

The component should not be created automatically by the actor.

Missing components should be treated as designer setup issues only for actors that are expected to participate in FOW.

For promoted map objects, the best shared location is `AWorldMapObject`, because enemy, mission, neutral, and player objects all derive from it.

For connections and anchors, add the same pattern only if they need direct FOW participation.

---

# New manager actor: `AWorldFowManager`

Create a new actor named:

`AWorldFowManager`

This actor owns all global world campaign FOW logic.

It should be separate from the regular RTS `FowManager`.

The manager should be responsible for:

* reading the current generated campaign state;
* registering FOW participants;
* resolving FOW state for anchors, connections, and world objects;
* updating `UWorldMapFowComponent` states;
* generating the CPU-side mask texture;
* pushing the mask texture to the cloud cube material;
* providing debug tools.

`AWorldFowManager` should not own campaign generation.

It should read from `AGeneratorWorldCampaign`, then calculate world campaign FOW.

---

# Single public FOW update call

`AWorldFowManager` should expose one main public update call:

`UpdateFowFromCurrentWorldState`

This is the high-level entry point for refreshing world campaign FOW.

Other systems should not directly update large parts of the FOW state.

When the world state changes, they should call:

`UpdateFowFromCurrentWorldState`

That call should:

1. read the current campaign world state;
2. read anchors and connections from `AGeneratorWorldCampaign`;
3. read promoted world objects through their owning anchors and anchor keys;
4. calculate which actors should be `NotVisible`, `Explorable`, `Visible`, or `POIVisible`;
5. update all cached `UWorldMapFowComponent` states;
6. update primary-click and outline eligibility indirectly through those states;
7. rebuild or update the CPU FOW mask;
8. push the mask texture to the cloud cube material.

This keeps FOW changes centralized and prevents systems from drifting out of sync.

---

# When to call `UpdateFowFromCurrentWorldState`

The update call should be used at predictable world campaign moments, for now only after the world is generated (not turns 
are implemented yet.)

so: after the campaign world has been generated;

The FOW manager should not perform per-frame FOW calculations.

The cloud material can still animate every frame, but the authoritative FOW state only changes through `UpdateFowFromCurrentWorldState`.

---

# Spawning `AWorldFowManager` from `AWorldPlayerController`

`AWorldPlayerController` should spawn the world FOW manager during `BeginPlay`.

Add a designer-set class reference on `AWorldPlayerController`:

`WorldFowManagerClass`

This should be a Blueprint-derived class based on `AWorldFowManager`.

Recommended flow:

1. `AWorldPlayerController::BeginPlay` starts.
2. Existing world generator setup runs.
3. Existing world menu setup runs.
4. Existing campaign settings setup runs.
5. Existing generate/restore world flow runs.
6. `AWorldPlayerController` spawns `WorldFowManagerClass`.
7. Controller passes itself and its `AGeneratorWorldCampaign` reference into the manager.
8. Manager initializes its actor/component lookups.
9. Manager calls `UpdateFowFromCurrentWorldState`.
10. Existing initial world setup completion can continue.

The controller should cache the spawned manager in a weak pointer or object pointer, depending on ownership expectations.

Recommended controller-facing settings:

| Setting                     | Purpose                                            |
| --------------------------- | -------------------------------------------------- |
| `WorldFowManagerClass`      | Blueprint class spawned at BeginPlay.              |
| `WorldFowCloudTargetActor`  | Actor containing the large cloud cube/static mesh. |
| `WorldFowCloudMaterialSlot` | Material slot receiving the generated mask.        |
| `WorldFowMaskParameterName` | Texture parameter used by the cloud material.      |
| `WorldFowMapBoundsSource`   | Bounds source for world-to-mask UV mapping.        |
| `bEnableWorldFowDebug`      | Enables debug tools.                               |

make sure that all those defaults settings that are for the fow cloud target actor are in the same category for easy access.
---

# Runtime FOW model

`AWorldFowManager` should build runtime records from the generated campaign state.

## Anchor records

Each `AAnchorPoint` should have a runtime FOW record.

Suggested data:

* anchor pointer;
* anchor key;
* cached FOW component, if the anchor participates directly;
* current FOW state;
* connected anchors;
* connected connections;
* promoted world object;
* reveal radius;
* dirty mask area.

## Connection records

Each `AConnection` should have a runtime FOW record.

Suggested data:

* connection pointer;
* connected anchors;
* current FOW state;
* corridor width;
* falloff;
* dirty mask area.

## World object records

Each `AWorldMapObject` and subclass should have a runtime FOW record.

Suggested data:

* object pointer;
* object class;
* owning anchor;
* anchor key;
* cached FOW component;
* current FOW state;
* POI settings;
* dirty mask area.

The anchor key should be the main relationship between a promoted object and its FOW state.

---

# Initial reveal rules

When `AWorldFowManager` initializes and calls `UpdateFowFromCurrentWorldState` for the first time:

1. Clear relevant FOW states to `NotVisible`.
2. Find the player HQ anchor from `AGeneratorWorldCampaign`.
3. Set the player HQ `AAnchorPoint` to `Visible`.
4. Set the player HQ promoted `AWorldPlayerObject` to `Visible`.
5. Find anchors connected to the player HQ.
6. Set those connected anchors to `Explorable`.
7. Set their promoted `AWorldMapObject` actors to `Explorable`.
8. Set connections between visible and explorable anchors to `Explorable`.
9. Evaluate POI-capable actors.
10. Set eligible tall landmark actors to `POIVisible`.
11. Generate the first mask texture.
12. Push the mask texture to the cloud cube material.

This gives the player:

* one visible starting area;
* several explorable frontier options;
* hinted routes toward those options;
* optional tall landmarks above the cloud.

---

# Progression reveal rules
Make sure that if later calls to the fow update happen the following is taken into account
Expected behavior:

* already visible anchors remain visible;
* newly reached or completed anchors become visible;
* directly connected hidden anchors become explorable;
* promoted objects on visible anchors become visible;
* promoted objects on explorable anchors become explorable;
* connections between visible anchors become visible;
* connections from visible anchors to explorable anchors become explorable;
* the mask texture is refreshed.

State should generally not downgrade during normal campaign exploration!!

---

# Primary-click gating

`AWorldPlayerController` should continue to own primary-click routing.

The FOW system should add a gate before type-specific click handling.

Recommended primary-click flow:

1. Controller receives primary click.
2. Controller gets the actor under the cursor.
3. Controller checks whether the actor implements `IWorldFowParticipant`.
4. If it does, controller asks `GetCanPrimaryClickInteract`.
5. If primary-click interaction is not allowed, click handling stops.
6. If primary-click interaction is allowed, existing click flow continues.

Existing type-specific handlers can remain:

* enemy object handler;
* mission object handler;
* player object handler;
* neutral object handler.

FOW only decides whether the primary click is allowed.

FOW should not decide what the click does after it is allowed.

---

# Outline gating

`UPlayerWorldOutliner` should use `IWorldFowParticipant` to decide whether an actor can be outlined.

Recommended outline flow:

1. Outliner receives the hovered actor.
2. Outliner checks whether the actor implements `IWorldFowParticipant`.
3. If it does, outliner calls `GetCanBeOutlined`.
4. `GetCanBeOutlined` delegates to the cached `UWorldMapFowComponent`.
5. The component checks the current `EWorldMapFowState`.
6. If outline is allowed, the actor can be outlined.
7. If outline is not allowed, no outline is shown.

This prevents hidden actors from being outlined just because they are physically under the cursor.

---

# World object behavior

Promoted world objects should follow their owning anchor by default.

Default mapping:

| Owning anchor state | World object state |
| ------------------- | ------------------ |
| `NotVisible`        | `NotVisible`       |
| `Explorable`        | `Explorable`       |
| `Visible`           | `Visible`          |

Special rule:

A POI-capable world object can become `POIVisible` even when its owning anchor is not visible.

This is useful for tall buildings, landmarks, enemy structures, towers, or major objectives that should be partially visible above the cloud.

`POIVisible` does not reveal the owning anchor.

---

# Connection behavior

Connections should visually communicate campaign route information.

Recommended connection state rules:

| Connected anchor state                    | Connection state        |
| ----------------------------------------- | ----------------------- |
| both anchors hidden                       | `NotVisible`            |
| one anchor visible, one anchor explorable | `Explorable`            |
| both anchors visible                      | `Visible`               |
| route only points toward a POI            | optional POI-style hint |

A visible anchor should not appear as a lonely hole in the fog.

The route from a visible anchor to its explorable neighbors should usually be hinted with a soft corridor reveal.

---

# Tall POI behavior

Tall POI objects are special FOW participants.
A POI actor should use `UWorldMapFowComponent` with POI behavior enabled.

Suggested POI settings:

| Setting                          | Purpose                                                |
| -------------------------------- | ------------------------------------------------------ |
| `bCanBecomePOIVisible`           | Allows this actor to enter `POIVisible`.               |
| `POIRevealRadius`                | Small mask radius around the POI.                      |
| `POIRevealFalloff`               | Softness of the POI mask.                              |
| `POIRevealOrigin`                | Optional origin override for the top of the structure. |
| `bPOIVisibleRevealsOwningAnchor` | Default false.                                         |
| `POIMarkerStyle`                 | Beacon, glow, silhouette, icon, or outline-only.       |

Important POI rules:

* `POIVisible` allows outline only;
* `POIVisible` does not allow primary-click interaction by default;
* `POIVisible` does not reveal the owning `AAnchorPoint`;
* `POIVisible` should contribute to the POI mask channel, not the full visible channel.

---

# Cloud mask texture

`AWorldFowManager` owns the generated FOW mask texture.

The mask is generated on the CPU because the system updates only at predictable campaign moments.
The large cloud cube material samples this mask and combines it with animated cloud visuals.

## Recommended mask channels

Use an RGBA mask.

| Channel | Meaning                                          |
| ------- | ------------------------------------------------ |
| Red     | Visible reveal amount.                           |
| Green   | Explorable/frontier reveal amount.               |
| Blue    | POI reveal amount.                               |
| Alpha   | Empty for now. |

This gives the material enough information to style each state differently.

Examples:

* `Visible` strongly clears cloud;
* `Explorable` thins or shapes the cloud as a frontier hint;
* `POIVisible` creates a small glow/opening/silhouette around landmarks;
* `NotVisible` remains covered.

---

# CPU mask generation

The mask should be generated from simple stamps.

## Anchor stamp

Used for `AAnchorPoint`.

Shape:

* circular or soft organic reveal;
* radius comes from the anchor/component state settings;
* falloff comes from the anchor/component state settings.

Used by:

* `Visible` anchors;
* `Explorable` anchors.

## Connection stamp

Used for `AConnection`.

Shape:

* capsule or soft line between connected anchors;
* width comes from the connection/component state settings;
* falloff controls softness around the route.

Used by:

* visible routes;
* explorable route hints.

## POI stamp

Used for POI-capable actors.

Shape:

* small circular or vertical-looking reveal around the POI origin;
* usually writes to the POI channel;
* should not clear a large terrain area.

Used by:

* `POIVisible` landmarks;
* tall structures;
* distant objective hints.



# Material behavior

Suggested material interpretation:

| Mask Channel | Material result                             |
| ------------ | ------------------------------------------- |
| Red          | Strong cloud removal.                       |
| Green        | Soft frontier thinning.                     |
| Blue         | POI glow, silhouette, or small cloud break. |
| Alpha        | Optional density/transition control.        |

The material should not decide gameplay state.

It only visualizes the mask produced by `AWorldFowManager`.

---

# State authority

`AWorldFowManager` is authoritative for world campaign FOW state.

`UWorldMapFowComponent` stores and exposes actor-local FOW state.

`IWorldFowParticipant` provides safe access to the cached FOW component and FOW-derived behavior checks.

`AWorldPlayerController` uses the interface/component to gate primary-click interaction.

`UPlayerWorldOutliner` uses the interface/component to gate outline behavior.

The generated mask texture is visual output only.

---

# Designer workflow

## Campaign generation workflow

Designers continue to use the existing world campaign generation flow.

`AGeneratorWorldCampaign` remains responsible for:

* anchor generation;
* connection generation;
* player HQ placement;
* enemy HQ placement;
* enemy object placement;
* neutral object placement;
* mission placement;
* promoted world object creation.

## FOW setup workflow

Designers configure:

1. `BP_WorldFowManager`, derived from `AWorldFowManager`.
2. `WorldFowManagerClass` on `AWorldPlayerController`.
3. Cloud cube actor/material reference.
4. Mask texture parameter name.
5. Map bounds source.
6. Default FOW reveal settings.
7. Optional actor-specific FOW settings.
8. Optional POI settings for tall landmarks.

## Actor setup workflow

Designers add `WorldMapFow` components in derived Blueprints.

Recommended component placement:

* on `AWorldMapObject`-derived Blueprints that should participate in FOW;
* on `AAnchorPoint` Blueprints if anchors need direct debug or visual FOW behavior;
* on `AConnection` Blueprints if connections need direct route FOW behavior;
* on tall POI actors that should become `POIVisible`.

The C++ class should cache the component during `PostInitializeComponents`.

---

# Debugging tools

`AWorldFowManager` should expose debug tools for designers.

Recommended debug actions:

* force `UpdateFowFromCurrentWorldState`;
* reveal all;
* hide all;
* show only `Explorable`;
* show only `POIVisible`;
* show player HQ reveal;
* rebuild mask texture;
* draw anchor FOW states;
* draw connection FOW states;
* draw reveal radii;
* draw connection corridor widths;
* preview mask channels;
* validate missing FOW components;
* validate actors that implement `IWorldFowParticipant` but have no cached component;
* validate duplicate anchor keys;
* validate world objects without owning anchors;
* validate connections with missing anchors;
* list primary-click interactable actors;
* list outlineable actors;
* list blocked `NotVisible` actors.

Debugging should make it clear why an actor can or cannot be clicked or outlined.
